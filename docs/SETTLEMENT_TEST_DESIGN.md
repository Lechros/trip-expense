# 정산(Settlement) 테스트 설계

정산 API(`GET /trips/:tripId/settlement`)와 계산 로직을 검증하기 위한 테스트 시나리오 설계입니다.  
원하는 동작이 있으면 각 시나리오별로 “이럴 때 이렇게 나와야 한다”를 정리해 주시면, 그에 맞춰 구현/테스트를 맞출 수 있습니다.

---

## 1. 입력 데이터 정의

정산 계산에 쓰이는 데이터는 다음과 같습니다.

| 데이터 | 설명 |
|--------|------|
| **Trip** | `baseCurrency`(기본 KRW), `additionalCurrency`(예: JPY) |
| **Members** | 멤버 목록 (id, displayName, userId / guestId) |
| **Entries** | 정산 항목. `paidBy`, `amount`, `currency`, `paidAt`, `beneficiaries`(수혜자 N명) |
| **ExchangeRecords** | 환전 기록. 결제자별·통화별 `rate`, `exchangedAt` (환율: 1 외화 = rate 원) |

- 항목이 **KRW**이면 환율 없이 그대로 원화로 사용.
- 항목이 **외화**(예: JPY)이면, **결제자(paidBy)**의 **paidAt에 가장 가까운** 환전 기록의 환율로 원화 환산.
- 해당 결제자·통화에 대한 환전 기록이 **없으면** 그 항목은 정산에서 **제외**하고, `excludedMessages`에 안내 문구 추가.

---

## 2. 출력 데이터 정의

API 응답에서 검증할 수 있는 항목입니다.

| 필드 | 설명 |
|------|------|
| **exchangeWarning** | 제외된 항목이 하나라도 있으면 true |
| **excludedMessages** | 제외된 항목에 대한 안내 문구 배열 |
| **memberSummaries** | 멤버별 `paidKrw`, `usedKrw`, `diffKrw` (결제 합계 − 사용 합계) |
| **transfers** | 이체 목록: "A가 B에게 amountKrw 원 전달" (최소 이체 횟수 그리디) |
| **myReceive** / **mySend** | 요청한 사용자(회원/게스트) 기준으로 받을/줄 이체 목록 |
| **mySummary** | 요청한 사용자의 정산 요약 (또는 null) |

---

## 3. 테스트 시나리오 목록

아래 시나리오는 “이런 입력이면 이렇게 나와야 한다”를 정할 때 쓰는 **체크리스트**입니다.  
각 시나리오에 대해 원하는 로직(기대값)을 정해 주시면, 그에 맞춰 구현·테스트를 작성하겠습니다.

### 3.1 인증·권한

| # | 시나리오 | 예상 동작 (확정 시 채움) |
|---|----------|--------------------------|
| S1 | 인증 없이 `GET /trips/:tripId/settlement` | 401 |
| S2 | 해당 여행 멤버가 아닌 사용자가 요청 | 403 |
| S3 | 존재하지 않는 tripId | 404 |
| S4 | 해당 여행 멤버(회원 또는 게스트)로 요청 | 200, 본인용 mySummary / myReceive / mySend 포함 |

---

### 3.2 환율·원화 환산

| # | 시나리오 | 예상 동작 (확정 시 채움) |
|---|----------|--------------------------|
| E1 | 항목이 모두 KRW | 환율 조회 없이 amount 그대로 paidKrw/usedKrw에 반영 |
| E2 | 항목이 JPY, 결제자에게 JPY→KRW 환전 기록 1건 (예: rate=9.4, exchangedAt < paidAt) | 해당 항목 금액 × 9.4 로 원화 환산 |
| E3 | 결제자에게 paidAt **이전**·**이후** 환전 기록이 둘 다 있음 | “가장 가까운” 한 건만 쓸지, 여러 건 평균할지 (원하는 규칙 명시) |
| E4 | 같은 결제자·같은 통화로 exchangedAt 다른 환전 기록이 여러 건 | 어떤 기록을 쓸지(가장 가까운 1건 vs 평균 등) (원하는 규칙 명시) |
| E5 | 항목 통화가 JPY인데, 결제자에게 JPY 환전 기록 없음 | 해당 항목 제외, excludedMessages에 “OO님의 JPY 환전 기록이 없어…” 추가, exchangeWarning true |

---

### 3.3 멤버별 합계 (paidKrw, usedKrw, diffKrw)

| # | 시나리오 | 예상 동작 (확정 시 채움) |
|---|----------|--------------------------|
| M1 | 정산 항목 없음 | 모든 멤버 paidKrw=0, usedKrw=0, diffKrw=0 |
| M2 | A가 30,000원 결제, 수혜자 A,B,C (3명) | A paidKrw +30,000 / A usedKrw +10,000 / B,C usedKrw 각 +10,000 |
| M3 | A가 30,000원 결제, 수혜자 A만 (1명) | A paidKrw +30,000, usedKrw +30,000, diffKrw 0 |
| M4 | A가 10,000원, B가 20,000원 결제, 수혜자 모두 A,B | 각자 paid 1만/2만, used 각 15,000 → A diff -5,000, B diff +5,000 |
| M5 | 소수점 있는 금액(또는 환산 결과 소수) | 반올림/버림 규칙 (소수 둘째 자리? 정수만?) (원하는 규칙 명시) |

---

### 3.4 이체 목록 (transfers) 생성

| # | 시나리오 | 예상 동작 (확정 시 채움) |
|---|----------|--------------------------|
| T1 | diffKrw가 모두 0 | transfers 빈 배열 |
| T2 | A가 50,000 받기, B가 50,000 주기 | transfers: [ { from: B, to: A, amountKrw: 50000 } ] |
| T3 | A +60,000 / B -40,000 / C -20,000 | 최소 이체: B→A 40,000, C→A 20,000 (또는 원하는 조합 명시) |
| T4 | A +30,000 / B -20,000 / C -10,000 | 예: B→A 20,000, C→A 10,000 (또는 다른 순서/조합 명시) |
| T5 | diffKrw 합이 0이 아님 (반올림 오차 등) | 허용 여부, 허용 시 어떻게 보정할지 (원하는 규칙 명시) |

---

### 3.5 제외·특수 케이스

| # | 시나리오 | 예상 동작 (확정 시 채움) |
|---|----------|--------------------------|
| X1 | 일부 항목만 환전 없음으로 제외 | 제외된 항목만 빠지고, 나머지로만 memberSummaries·transfers 계산 |
| X2 | soft-deleted 항목(deletedAt not null) | 정산 계산에 포함하지 않음 |
| X3 | 수혜자 0명인 항목 | 해당 항목 처리 방식 (제외? paid만 반영? 등) (원하는 규칙 명시) |
| X4 | 결제자(paidBy)가 현재 멤버 목록에 없음(탈퇴 등) | paidKrw에 포함하지 않거나, “알 수 없음” 등 처리 (원하는 규칙 명시) |

---

## 4. 테스트 구현 방식 제안

- **백엔드**: `backend/src/routes/settlement.test.ts`에서 `buildApp`, `prisma`로 Trip/Members/Entries/ExchangeRecords를 생성한 뒤 `GET /trips/:tripId/settlement`를 호출하고, 위 필드를 검증.
- **시나리오별**: `describe('정산 API - 환율 적용')`, `describe('정산 API - 멤버별 합계')`, `describe('정산 API - 이체 목록')` 등으로 나누고, 위 표의 시나리오 번호(S1~T5, E1~E5, M1~M5, X1~X4)를 it 제목에 넣으면 나중에 “E3는 이렇게 동작해야 해”처럼 맞추기 쉽습니다.

원하시는 동작을 시나리오 번호와 함께 알려주시면, 그에 맞춰 실제 테스트 케이스(기대값)와 필요 시 정산 로직을 수정하겠습니다.
