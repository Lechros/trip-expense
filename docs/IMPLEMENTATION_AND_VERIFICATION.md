# 기능 구현 방안 · 검증 방안

SPEC §13 마일스톤과 성공 기준을 기준으로, **어떻게 구현할지**와 **어떻게 검증할지**를 정리한다.

---

## 1. 기능 구현 방안

### 1.1 전체 전략

- **API 우선**: 백엔드 API를 먼저 구현하고, 프론트는 목업을 해당 API 호출로 교체한다. 타입·스키마는 가능하면 공유(Zod 스키마 또는 generated 타입)하여 일관성을 유지한다.
- **단계별 연동**: M2 완료 후 프론트 여행 목록/참여 연동 → M3 완료 후 지출 탭 연동 → M4 후 환전·정산 탭 연동 → M5에서 설정·게스트·배포.
- **접근 제어 일원화**: `trips/:tripId` 하위 모든 API에서 “요청자가 해당 trip 멤버인지” 검증하는 미들웨어/헬퍼를 두고, 권한(owner만 수정 등)은 라우트별로 추가 검증한다.

### 1.2 M2 — 여행·멤버

| 작업 | 내용 |
|------|------|
| 백엔드 | `GET/POST /trips`, `GET/PATCH /trips/:id`, `GET /trips/join-info`, `POST /trips/join`, `GET /trips/:tripId/members`, (owner만) `PATCH/DELETE .../members/:memberId`. TripMember에 displayName, colorHex. |
| 접근 제어 | 회원: JWT에서 userId. 요청자가 trip 멤버인지 확인하는 `requireTripMember(tripId)` 같은 헬퍼. join 시 회원/게스트 분기. |
| 프론트 | 로그인 상태 저장(Zustand 또는 Context), API 클라이언트(Authorization 헤더). 여행 목록 = `GET /trips`, 여행 생성/참여 화면을 API와 연동. |

### 1.3 M3 — 정산 항목

| 작업 | 내용 |
|------|------|
| 백엔드 | `GET/POST /trips/:tripId/entries`, `PATCH/DELETE .../entries/:id` (DELETE는 soft delete). query로 결제자/수혜자/날짜/통화 필터. paidBy·beneficiaries가 해당 trip 멤버인지 검증. |
| 프론트 | 지출 탭: 목업 제거, `GET /trips/:tripId/entries`로 목록, 추가/상세/수정/삭제를 해당 API로 수행. 필터 파라미터 연동. |

### 1.4 M4 — 환전·정산 계산

| 작업 | 내용 |
|------|------|
| 백엔드 | 환전: `GET/POST/PATCH/DELETE /trips/:tripId/exchanges`. 정산: `GET /trips/:tripId/settlement` — 유효 항목 기준 통화 환산, 멤버별 차이, 최소 이체 리스트 반환. 환전 부족 시 해당 항목 제외 + 메타 정보로 안내. |
| 프론트 | 환전 탭·정산 탭을 위 API 응답으로 렌더링. |

### 1.5 M5 — 설정·마무리

| 작업 | 내용 |
|------|------|
| 백엔드 | 여행 설정 수정(owner만), 멤버 내보내기(owner만). 게스트 참여 플로우: join-info(여행 비밀번호 검증) → join(회원/기존 게스트/신규 게스트). |
| 프론트 | 설정 탭 API 연동. 참여 플로우(링크+비밀번호)를 SPEC §2.2에 맞게 구현. |
| 공통 | 보안 체크리스트(PROJECT_PLAN §4.2) 점검, rate limiting·로그 정책, Vercel·Railway 배포. |

### 1.6 공통 구현 원칙

- **입력 검증**: 모든 API body/query는 백엔드에서 Zod(또는 동일 스키마)로 재검증. DB 제약과 병행.
- **금액**: Decimal/정수 원 단위로 저장·계산, 부동소수 오차 방지.
- **타입 공유**: API 요청/응답 타입을 프론트에서 재사용하거나, OpenAPI/스키마 생성으로 일치시킨다.

## 2. 검증 방안

### 2.1 단위·통합 검증 (백엔드)

| 구분 | 대상 | 도구/방식 | 목적 |
|------|------|-----------|------|
| 단위 | 비즈니스 로직(정산 계산, 환율 적용 등) | Jest 또는 Vitest | 순수 함수·서비스 로직이 SPEC과 일치하는지 검증. |
| API 테스트 | 라우트별 요청/응답 | Supertest + Fastify inject, 또는 Vitest + fetch | 인증 없음/잘못된 tripId 시 401·403, body 검증 실패 시 400, 정상 시 200/201 및 스키마 일치. |
| DB | 마이그레이션·제약 | `prisma migrate dev` 후 스키마 검증, 필요 시 시드 데이터로 제약·FK 확인. | 스키마가 SPEC과 맞고, 잘못된 데이터가 들어가지 않음을 확인. |

추천 순서: 정산 계산 로직 등 핵심 알고리즘 단위 테스트 → API 통합 테스트(인증·권한·CRUD) → 필요 시 E2E.

### 2.2 E2E·수동 검증 (프론트 + API)

| 구분 | 내용 |
|------|------|
| E2E | Playwright 등으로 “로그인 → 여행 생성 → 지출 추가 → 정산 탭에서 결과 확인” 같은 핵심 플로우 자동화. (선택, 초기에는 수동으로 대체 가능.) |
| 수동 QA | SPEC §14 성공 기준 기준 체크: 모바일에서 여행 생성·참여, 지출 입력·수정·삭제, 정산 결과 “누가 누구에게 얼마”, 환전 부족 안내, 미참여 여행 미노출. |
| 접근성 | 주요 화면 키보드·스크린리더 사용 가능 여부, 터치 타겟·contrast 등 간단 점검. |

### 2.3 보안·운영 검증

- PROJECT_PLAN §4.2 체크리스트: TripMember 기반 접근 제어, paidBy/beneficiaries 검증, owner만 설정·멤버 관리, rate limiting, 로그에 금액·개인정보 미포함.
- 배포 후: Vercel·Railway 로그 및 에러 모니터링, 필요 시 헬스 체크(`/health`) 연동.

### 2.4 검증 적용 우선순위

1. **필수**: API 통합 테스트(인증·권한·CRUD), 수동으로 성공 기준 충족 여부 확인.
2. **권장**: 정산 계산 등 핵심 로직 단위 테스트, 보안 체크리스트 점검.
3. **선택**: E2E 자동화, 접근성 정식 감사.

---

## 3. 요약

- **구현**: API 우선, M2→M3→M4→M5 순으로 단계 구현 후 프론트 목업을 API로 교체. 접근 제어와 입력 검증을 백엔드에서 일원화.
- **검증**: 백엔드는 단위(핵심 로직) + API 통합 테스트, 프론트는 수동 QA(성공 기준) 필수. 보안 체크리스트와 배포 후 모니터링으로 안정성 확보.
