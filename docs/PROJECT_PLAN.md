## 1. 프로젝트 개요

### 1.1 목적

친구들과의 여행 중 발생한 지출을 기록하고, 여행 종료 후 누가 누구에게 얼마를 정산해야 하는지 계산·표시하는 웹 애플리케이션.

### 1.2 핵심 가치

- **모바일 우선**: 여행 중 손쉽게 입력·확인 가능한 UI/UX
- **간편한 시작**: 회원가입 최소화 또는 비회원 사용 가능성 검토
- **투명한 정산**: 결제자·수혜자·환율이 명확히 기록·계산됨
- **보안**: 금액·정산 데이터에 대한 검증과 접근 제어 강화

### 1.3 제약·가정

- 초대규모 트래픽 대응 불필요 (소규모 팀/그룹 사용 가정)
- 배포는 무료 티어 위주 (Vercel, Railway 등)
- Next.js API는 SSR/정적 목적만, 비즈니스 로직·계산은 별도 백엔드에서 수행

## 2. 기능 요구사항 (High-level → 상세)

### 2.1 인증 및 사용자

| 구분         | 요구사항       | 상세 설계                                                                                                                                                                                                                  |
| ------------ | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **회원가입** | 아주 쉽게 가능 | 이메일+비밀번호 또는 소셜(Google 등) 1클릭. **결론**: 이메일 인증은 초기 버전에서 생략                                                                                                                                         |
| **비회원**   | 가능하면 지원  | **옵션 A**: 비회원은 “여행 링크+참여 코드”로만 접근, 정산 기록·계산은 가능하나 “내 정산 요약” 등 개인화 제한. **옵션 B**: 모든 핵심 기능은 회원만 사용. 권장: **옵션 A**로 시작해, 링크/코드 유출 리스크 안내 후 회원 유도 |
| **세션**     | 일관된 경험    | **결론**: JWT(access + refresh). 무료 티어에서 Redis 불필요, 구현 단순. refresh로 장기 여행 중 끊김 최소화                                                                                                                                                              |

**권장 플로우**

- **회원**: 이메일(또는 소셜) 가입 → 로그인 → 여행 생성/참여 → 모든 기능 사용.
- **비회원(선택)**: “여행 참여 링크 + 6자리 코드” 입력 → 해당 여행에만 “게스트”로 참여(이름·색상만 설정), 데이터는 해당 여행 범위로만 한정. 게스트는 정산 항목 입력·확인·정산 계산은 가능, 여행 생성·멤버 추방 등은 불가.

| 구분                     | 회원     | 비회원(게스트)       |
| ------------------------ | -------- | -------------------- |
| 여행 생성                | ✅       | ❌                   |
| 링크/코드로 참여         | ✅       | ✅                   |
| 정산 항목 입력/수정/삭제 | ✅       | ✅                   |
| 환전 기록 입력           | ✅       | ✅                   |
| 정산 계산 보기           | ✅       | ✅                   |
| 여행 설정·멤버 관리      | 생성자만 | ❌                   |
| 여러 여행 목록           | ✅       | 링크로 들어간 여행만 |

### 2.2 여행(Trip) 관리

| 항목          | 내용                                                                                                 |
| ------------- | ---------------------------------------------------------------------------------------------------- |
| **생성**      | 회원만 여행 생성 가능. 이름, 기간(시작일–종료일), 국가(ISO country code) 필수.                       |
| **참여**      | “여행 참여 링크” 또는 “초대 코드”로 기존 여행에 조인. 내가 속한 여행만 목록에 노출                   |
| **기본 통화** | 원(KRW)을 기본 통화로 고정                                                                           |
| **추가 통화** | 여행 국가 통화(예: JPY, USD) 1개 기본 추가. **결론**: 최대 1개 추가 통화만 허용(즉 baseCurrency + 추가 1개 = 최대 2통화). 복잡도·정산 계산 단순화.             |
| **접근 제어** | 모든 API에서 “현재 사용자 ↔ 여행 멤버십” 검증; 미참여 여행 데이터는 절대 반환하지 않음               |

**데이터 예시**

- `Trip`: id, name, description(선택), startDate, endDate, countryCode, baseCurrency(KRW), additionalCurrencies[], createdAt, createdByUserId
- `TripMember`: tripId, userId(또는 guestId), displayName, colorHex, role(owner/member/guest), joinedAt

### 2.4 정산 항목(Settlement Entry) — 핵심 도메인

**의미**

- “누가, 언제, 얼마를 결제했고, 그 비용이 누구 것인지”를 기록하는 단위. 실제 송금/결제는 하지 않고 **기록만** 유지.

**필드(후보)**

| 필드              | 타입                      | 설명                                                                 |
| ----------------- | ------------------------- | -------------------------------------------------------------------- |
| id                | UUID                      | 항목 식별자                                                          |
| tripId            | FK                        | 소속 여행                                                            |
| **paidBy**        | userId (또는 guestId)     | 실제로 결제한 사람(돈 낸 사람)                                       |
| **beneficiaries** | userId[] (또는 guestId[]) | 그 돈으로 혜택 본 사람(누구 것을 낸 건지). 1:N 가능(예: 식비 3명 분) |
| amount            | decimal                   | 결제 금액                                                            |
| currency          | code                      | KRW, JPY 등                                                          |
| **recordedBy**    | userId 또는 guestId       | 이 항목을 입력한 사람(감사/추적용). **결론**: 회원이면 userId, 게스트면 guestId 저장. |
| **paidAt**        | datetime                  | 실제 결제가 발생한 시각(사용자 입력)                                 |
| **recordedAt**    | datetime                  | 시스템에 입력된 시각(서버 저장, 수정 불가 아님)                      |
| memo              | string(선택)              | 메모(예: “점심 라멘”)                                                |
| **deletedAt**     | datetime?                 | soft delete; null이면 유효 항목                                      |

**비즈니스 규칙**

- amount > 0, currency는 해당 여행에서 허용된 통화만.
- paidBy, beneficiaries 모두 해당 여행의 멤버여야 함.
- **결론**: paidAt은 여행 기간으로 제한하지 않음(선입력·사후 보정 허용).

**모바일 UX**

- 목록: **결론** 리스트형(모바일에서 항목 수 많을 때 스캔·가상 리스트에 유리). 스크롤·가상화로 수십 개 부드럽게 표시.
- 필터: 결제자/수혜자/날짜/통화.
- 수정: 항목 탭/클릭 → 상세 또는 바텀시트에서 수정.
- **Soft delete**: “삭제” 시 `deletedAt` 설정, 목록에서는 기본 비표시(필터로 “삭제된 항목 보기” 옵션 가능). 정산 계산 시 `deletedAt != null` 항목 제외.

### 2.5 환전 기록(Exchange Record)

- **목적**: 정산 계산 시 “해당 결제자가 사용한 환율”을 산출하기 위함.
- **입력 예**: “1 JPY = 9.4 KRW 로 10,000 JPY 환전” → 94,000 KRW 상당 기록.

**필드(후보)**

| 필드                  | 타입          | 설명                                             |
| --------------------- | ------------- | ------------------------------------------------ |
| id                    | UUID          | 식별자                                           |
| tripId                | FK            | 여행                                             |
| userId (또는 guestId) | FK            | 환전한 사람                                      |
| sourceCurrency        | code          | JPY                                              |
| targetCurrency        | code          | KRW (기본 통화)                                  |
| rate                  | decimal       | 1 sourceCurrency = rate targetCurrency (예: 9.4) |
| sourceAmount          | decimal       | 10000                                            |
| targetAmount          | decimal       | 94000. **결론**: 서버에서 rate×sourceAmount로 계산해 저장(일관성). |
| exchangedAt           | date/datetime | 환전한 날(또는 시각)                             |
| recordedAt            | datetime      | 입력 시각                                        |

**비즈니스 규칙**

- rate > 0, sourceAmount > 0.
- 정산 계산 시 “결제자별”로 해당 결제자의 환전 기록만 모아서 **평균 환율**로 사용(아래 정산 계산 참고).

---

### 2.6 정산 계산 화면

**표시 내용**

1. **통화별 요약(또는 기준 통화로 환산)**
   - 각 멤버가 “결제한 금액”(원화 환산)과 “사용한 금액”(원화 환산)을 계산.
2. **멤버별 차이**
   - (내가 낸 금액) − (내가 쓴 금액) = 양수면 “받을 돈”, 음수면 “낼 돈”.
3. **누가 누구에게 얼마 받아야 하는지**
   - 최소 이체 횟수로 정산하는 알고리즘(예: 그리디 정리) 결과를 문장/리스트로 표시.
   - 예: “A가 B에게 30,000원 받기”, “C가 A에게 15,000원 주기”.

**환율 적용**

- 각 **결제(정산 항목)** 에 대해:
  - 결제자(paidBy)의, 해당 **결제 시점(paidAt)** 에 가장 가까운 환전 기록들로 “평균 환율” 산출.
  - 해당 통화 → KRW(또는 baseCurrency) 환산 금액 = amount × (해당 평균 환율).
- **환율 부족 시**
  - 해당 결제자에게 해당 통화/기간의 환전 기록이 없으면:
    - “OO님의 JPY 환전 기록이 없어 이 결제는 원화로 환산할 수 없습니다” 등 안내.
    - 선택: 해당 항목을 계산에서 제외하고 경고만 표시, 또는 “수동 환율 입력” 허용(나중 단계).

**정산 계산 알고리즘(누가 누구에게)**

1. 모든 유효 정산 항목을 기준 통화(KRW)로 환산(위 환율 적용).
2. 멤버별로 “총 결제한 금액”과 “총 사용한 금액(beneficiary로 참여한 비용의 1/N)”을 계산.
3. 순 차이 = 결제한 금액 − 사용한 금액 → 양수면 받을 사람, 음수면 줄 사람.
4. **최소 이체 정산**: 받을 사람들(debtors)과 줄 사람들(creditors)을 그리디하게 매칭해 “A가 B에게 X원 전달” 형태의 리스트 생성. (예: Stack 기반 또는 우선순위 큐로 큰 금액부터 상쇄.)

**UI**

- 모바일: 스크롤 가능한 요약 카드 + “정산 결과” 리스트(누가 누구에게).
- 데스크톱: 같은 데이터를 테이블/카드로 확장.

### 2.7 설정 페이지

- **접근**: 여행 내 “설정” 또는 “톱니바퀴” 진입.
- **여행 설정(모두 또는 생성자만)**
  - 이름, 설명, 기간, 국가, 기본/추가 통화.
- **멤버 관리(생성자만)**
  - 멤버 목록, 내보내기(역할 변경 또는 제거). 생성자는 자신을 제거할 수 없거나, 제거 시 “다른 멤버에게 생성자 이전” 플로우 필요.
- **데이터 일관성**
  - 기간 변경 시 기존 정산 항목의 paidAt과 충돌하지 않도록 유효성 검사(선택); 멤버 제거 시 해당 멤버가 paidBy/beneficiaries인 항목은 “삭제된 멤버”로 표시하거나 정책에 따라 처리.

## 3. 기술 스택 및 아키텍처

### 3.1 Next.js 사용 여부

| 관점          | 결론                                                                                                     |
| ------------- | -------------------------------------------------------------------------------------------------------- |
| SSR/SEO       | 여행 정산 앱은 대부분 로그인 후 사용 → SEO 필수 아님. 정적 랜딩/로그인 페이지만 SSR 가능.                |
| 라우팅·번들링 | App Router + React 18, 코드 스플리팅 등은 Next.js 없이 Vite+React Router로도 가능.                       |
| API           | 요구사항대로 **비즈니스 로직·계산·DB 접근은 별도 백엔드**에서만 수행. Next.js API Route는 사용하지 않음. |
| 배포          | Vercel 배포 편의성은 Next.js와 잘 맞음.                                                                  |

**권장**: **Next.js 사용**

- 이유: Vercel 배포 최적화, App Router로 레이아웃/라우팅 단순화, 필요 시 랜딩/로그인 페이지만 SSR 가능. API는 Next에 두지 않고 **독립 백엔드(Railway)** 만 사용.

### 3.2 프론트엔드 스택

| 항목           | 선택                                                           | 비고                                                              |
| -------------- | -------------------------------------------------------------- | ----------------------------------------------------------------- |
| 프레임워크     | **Next.js 14+ (App Router)**                                   | React 18, TypeScript                                              |
| 스타일         | **Tailwind CSS**                                               | 모바일 우선 반응형                                                |
| UI 컴포넌트    | **shadcn/ui**                                                  | 조사 결과 적합; Radix 기반, 커스터마이즈 용이, 접근성·디자인 품질 |
| 상태·서버 상태 | **Zustand** (클라이언트) + **TanStack Query** (서버 상태·캐시) | 규칙에 부합                                                       |
| 검증           | **Zod**                                                        | 입력·API 스키마 일원화                                            |
| 폼             | React Hook Form + Zod                                          | 복잡 폼 대응                                                      |

**UI 조사 요약**

- shadcn: 무료, 모바일 친화, 다크모드 등 확장 쉬움 → **채택**.
- MUI/Chakra: 무거울 수 있음.
- 별도 디자인 시스템 없으면 shadcn으로 진행.

### 3.3 백엔드 스택

| 항목       | 선택                                            | 비고                                    |
| ---------- | ----------------------------------------------- | --------------------------------------- |
| 런타임     | **Node.js**                                     | 팀 친화성, TypeScript 공유              |
| 프레임워크 | **Fastify**                                     | **결론** 채택. 성능·스키마 검증 플러그인, TypeScript 친화. |
| 언어       | **TypeScript**                                  | 프론트와 타입·Zod 스키마 재사용 가능    |
| DB         | **PostgreSQL** (Railway 기본 제공)              | 관계형, 트랜잭션, 정산 로직에 적합      |
| ORM        | **Prisma**                                      | **결론** 채택. 마이그레이션·에코시스템·타입 안전성.       |
| 인증       | **JWT** (access + refresh)                      | **결론** 채택. 무료 티어에서 Redis 불필요.              |
| 검증       | **Zod**                                         | 요청 body/query 일관 검증               |

---

### 3.4 아키텍처 다이어그램(개념)

```
[사용자] ←→ [Next.js (Vercel)]
                ↓ HTTPS only
         [Backend API (Railway)]
                ↓
         [PostgreSQL (Railway)]
```

- 프론트는 **오직 백엔드 API**만 호출(Next API Route는 SSR/프록시 목적만 필요 시).
- CORS: 백엔드에서 Vercel 도메인(및 localhost)만 허용.
- 쿠키 사용 시 SameSite, Secure; JWT 사용 시 메모리 또는 httpOnly 쿠키 저장 시 XSS 완화.

## 4. 보안 및 검증

### 4.1 원칙

- **인증**: 모든 여행/정산 관련 API는 인증 필수(비회원 참여 시에는 “여행 토큰” 또는 제한된 API만 별도 설계).
- **권한**: 요청 시 `tripId`가 들어오면 “현재 사용자가 해당 여행 멤버인지”를 반드시 검증.
- **입력**: 모든 금액·날짜·외래키는 백엔드에서 Zod(또는 동일 스키마)로 재검증. DB 레벨 제약(NOT NULL, CHECK, FK) 병행.
- **금액**: decimal/정수 센트 방식으로 부동소수 오차 방지(예: 원화는 정수 원, 소수 없음).

### 4.2 체크리스트

- [ ] 회원/비회원 구분에 따른 라우트·API 접근 제어 정리
- [ ] TripMember 기반 접근 제어 미들웨어(또는 서비스 레이어) 적용
- [ ] 정산 항목 생성/수정/soft delete 시 paidBy, beneficiaries가 모두 해당 trip 소속인지 검증
- [ ] 환전 기록은 해당 trip 멤버만 입력/수정/삭제
- [ ] 설정 페이지: 멤버 제거·역할 변경은 생성자만 가능하도록 API에서 강제
- [ ] rate limiting(Railway 또는 미들웨어)으로 남용 방지
- [ ] 민감 로그에 금액·개인정보 포함 금지

## 5. 데이터 모델 요약

- **User**: id, email, passwordHash(또는 oauthId), createdAt
- **Guest**(비회원 시): id, tripId, displayName, colorHex, inviteCode, createdAt
- **Trip**: id, name, description, startDate, endDate, countryCode, baseCurrency, additionalCurrencies[], createdByUserId, createdAt
- **TripMember**: id, tripId, userId(nullable), guestId(nullable), displayName, colorHex, role(owner/member/guest), joinedAt
- **SettlementEntry**: id, tripId, paidByUserId/paidByGuestId, recordedByUserId(nullable), recordedByGuestId(nullable, 둘 중 하나 필수), amount, currency, paidAt, recordedAt, memo, deletedAt
- **SettlementBeneficiary**: entryId, userId 또는 guestId (N:N)
- **ExchangeRecord**: id, tripId, userId/guestId, sourceCurrency, targetCurrency, rate, sourceAmount, targetAmount, exchangedAt, recordedAt

(실제 구현 시 FK, 인덱스, unique 제약은 ERD 단계에서 정리.)

## 6. API 설계 (REST, 개요)

- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`
- `GET /me`, `PATCH /me`
- `GET /trips`, `POST /trips`, `GET /trips/:id`, `PATCH /trips/:id`, `POST /trips/join` (코드/링크로 참여)
- `GET /trips/:tripId/members`, `PATCH /trips/:tripId/members/:memberId`, `DELETE /trips/:tripId/members/:memberId` (생성자만)
- `GET /trips/:tripId/entries`, `POST /trips/:tripId/entries`, `PATCH /trips/:tripId/entries/:id`, `DELETE /trips/:tripId/entries/:id` (soft delete)
- `GET /trips/:tripId/exchanges`, `POST /trips/:tripId/exchanges`, `PATCH/DELETE ...`
- `GET /trips/:tripId/summary` 또는 `GET /trips/:tripId/settlement` → 정산 계산 결과(누가 누구에게 얼마) 반환

모든 `trips/:tripId` 하위 라우트에서 “현재 사용자(또는 게스트)가 해당 trip 멤버인지” 검증.

## 7. 배포

| 구분       | 서비스      | 용도                                                    |
| ---------- | ----------- | ------------------------------------------------------- |
| 프론트엔드 | **Vercel**  | Next.js 빌드·배포, 무료 티어 (대역·빌드 시간 제한 있음) |
| 백엔드     | **Railway** | Node API + PostgreSQL, 무료 크레딧/월 제한 내 사용      |
| 도메인     | 선택        | Vercel 기본 도메인 또는 커스텀                          |

- **무료 티어 참고**: Vercel은 개인 프로젝트 무료; Railway는 크레딧 기반이므로 DB+API 동시 운영 시 사용량 모니터링 필요. 초기에는 단일 PostgreSQL + 단일 API 서비스로 구성해 비용 최소화.
- 환경 변수: 프론트엔드 `NEXT_PUBLIC_API_URL`, 백엔드 `DATABASE_URL`, `JWT_SECRET`, `CORS_ORIGIN` 등.
- DB 마이그레이션: Railway 배포 시 자동 실행 또는 CI에서 실행하도록 설정.

## 9. 성공 기준

- 모바일에서 여행 생성·참여, 정산 항목 입력·수정·soft delete가 불편 없이 가능.
- 정산 계산 결과가 “누가 누구에게 얼마”로 명확히 나오고, 환전이 없는 결제는 안내됨.
- 내가 속하지 않은 여행 데이터는 노출되지 않음.
- 배포 후 Vercel + Railway 무료 범위 내에서 안정 동작.
